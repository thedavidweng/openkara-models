# OpenKara artifact catalog contract

This repository is the single source of truth for every model and ONNX Runtime
artifact consumed by the OpenKara app. The contract has three layers:

1. **Immutable release manifests** — one per infrastructure release, content-addressed,
   never mutated after publication.
2. **Stable-channel pointer** — a tiny document that advances only after every
   referenced manifest, asset, and release gate passes.
3. **`latest.json` migration adapter** — temporary, generated from the stable
   pointer, deleted once OpenKara PR #165 consumes the versioned schema
   (issue #18 PR 4).

OpenKara must validate the schema instead of embedding a second Rust-only
interpretation. It must reject incompatible target/runtime/model combinations
using the explicit compatibility edges rather than inferring from version
strings.

## Schemas

| Document               | Schema file                          | `schema_version`                  |
| ---------------------- | ------------------------------------ | --------------------------------- |
| Release manifest       | `catalog/schema-release-v1.json`     | `openkara.catalog/release-v1`     |
| Stable-channel pointer | `catalog/schema-channel-v1.json`     | `openkara.catalog/channel-v1`     |

Both are JSON Schema (draft 2020-12). Field-level structure is enforced by the
schema; cross-field invariants (referential integrity, immutable URLs,
target/provider consistency, generation monotonicity) are enforced by
`scripts/catalog_model.py` and invoked from `scripts/validate_catalog.py`.

## Validation

```bash
python scripts/validate_catalog.py catalog/releases/<release-id>.json
python scripts/validate_catalog.py --schema channel catalog/channels/stable.json
python scripts/validate_catalog.py --all-fixtures
```

`--all-fixtures` validates every document under `catalog/fixtures/valid/` (must
pass) and `catalog/fixtures/invalid/` (must fail, checked against the sibling
`<name>.expected.json` sidecar). Fixtures are generated by
`scripts/generate_catalog_fixtures.py`; never hand-edit them — edit the
mutation table and regenerate.

## Release manifest

A release manifest describes every model, runtime, and bundle artifact
published together, plus the compatibility edges and supply-chain references
that gate stable promotion. Required top-level fields:

- `schema_version`, `release_id` (`YYYY-MM-DD-NNN`), `generation` (monotonic),
  `created_at`, `producer` (repo + commit SHA + workflow + run id).
- `artifacts.{models,runtimes,bundles}` — typed artifact objects.
- `compatibility[]` — explicit model/runtime/target/provider edges.
- `supply_chain` — release-level SBOM, license, notice, provenance refs.
- `gates` — quality, performance, and compatibility report refs (issue #21).

### Artifact identity

Every artifact carries a stable `artifact_id` that **must not be reused for
different bytes**. Deprecation replaces an ID with a new one; it never silently
reuses the old ID. A deprecated artifact records `replacement_artifact_id`
(must exist in the same release) and an optional `sunset_generation`.

### Immutability

`download_url` and the channel `release_manifest_url` must be immutable HTTPS
URLs containing a release tag or content-addressed (40/64 hex) segment. Mutable
"latest"-style URLs are rejected. Every referenced file carries a byte size and
SHA-256 (`archive_digest` plus per-file `extracted_file_digests`).

### Model artifact

Declares `profile` (`balanced` / `quality` / `karaoke_2stem`), `target_triple`
(`null` for portable), `arch`, `os`, and a `model` block with: `format`
(`onnx` / `ort`), `opset`, `ir_version`, `tensor_interface` (`waveform` /
`spectral-core`), input/output semantics, `precision`, `segment_frames`,
`sample_rate`, `stem_profile`, `required_operator_config` (optional in v1 —
pre-#19 model releases may omit it; issue #19 PR 2 makes it required for new
releases and back-fills existing ones), and an explicit `compatible_runtime_ids`
list. The app resolves a runtime only through this list — never by parsing
version strings.

### Runtime artifact

Declares `target_triple`, `arch`, `os`, a `runtime` block (version, ORT C API
level, execution providers, companion files, deployment target,
`supported_model_artifact_ids`), and a `toolchain` block (image, compiler, SDK,
deployment target, CMake flags, submodule state). Built from an exact upstream
tag + commit + source lock (issue #19).

### Bundle artifact

Ties one model artifact to one runtime artifact for a specific
`target_triple`, `execution_provider`, `precision`, and `graph_format`. Its
target must match its runtime's target; its provider must be in the runtime's
providers. Only Pareto-optimal bundles are published (issue #22).

## Stable-channel pointer

`catalog/channels/stable.json` references exactly one immutable release
manifest by URL, SHA-256, and size. It advances in one atomic step after every
referenced asset and gate passes. `generation` is monotonically non-decreasing
and must equal the referenced manifest's generation.

## `latest.json` migration adapter

OpenKara PR #165 currently fetches an ad hoc `latest.json` of the form:

```json
{
  "htdemucs":     { "tag": "...", "url": "...", "sha256": "...", "size": 123 },
  "htdemucs_ft":  { "tag": "...", "url": "...", "sha256": "...", "size": 123 }
}
```

Issue #18 PR 2 generates this file from the stable pointer as a **migration
alias only**. It is deleted in issue #18 PR 4 once OpenKara #167 switches to
the versioned schema. It must not grow new fields or become a second contract.

## Invariants enforced

1. One catalog entry identifies one immutable asset by URL, size, and SHA-256.
2. The stable pointer never references a mutable asset or partial release.
3. Catalog generation is monotonic; `release_id` is unique.
4. Model/runtime compatibility is explicit; the app does not infer it.
5. Unknown required schema fields fail validation (`unevaluatedProperties: false`).
6. A deprecated artifact's replacement must exist in the same release.
7. Bundle target/provider must match its runtime.
8. `latest.json` is a migration alias only and is deleted after migration.
